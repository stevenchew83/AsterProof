# Inspinia Production Vendor Bundle Design

## Goal

Restore production JavaScript behavior for the Inspinia shell by shipping the
missing `inspinia/static/js/vendors.min.js` bundle as a tracked static asset.

## Context

Production currently breaks on authenticated dashboard pages even though local
development works. Reported symptoms include:

- sidenav toggle interactions not working
- DataTables-based pages failing to initialize

The relevant asset load order already exists in `inspinia/templates/base.html`:

- `js/config.js`
- `css/vendors.min.css`
- `css/app.min.css`
- `js/vendors.min.js`
- `js/app.js`

The app and plugin files are present in production, but the shared vendor bundle
is not.

## Root Cause

Production serves `http://18.142.249.181/static/js/app.js`,
`/static/js/config.js`, and the DataTables plugin files, but
`/static/js/vendors.min.js` returns `404`.

That bundle is generated by the existing frontend build from
`plugins.config.js` and currently contains:

- `jquery`
- `bootstrap.bundle`
- `lucide`
- `simplebar`

These are the shared dependencies expected by the Inspinia shell and parts of
the page-local JavaScript. Without `vendors.min.js`, production loads the theme
code in `app.js` but not the libraries it depends on.

The most likely reason the file is missing from production is that the repo
currently ignores minified JavaScript broadly via `.gitignore` (`*.min.js`),
which makes `inspinia/static/js/vendors.min.js` easy to build locally but easy
to omit from source control and deployment artifacts.

## Decision (approved)

Use **Option 1: track and ship `inspinia/static/js/vendors.min.js`**.

The production asset model stays the same:

- Django continues serving static assets from `/static/`
- `base.html` keeps loading local static files
- no CDN fallback is introduced
- no change is made to production static settings or template load order

Instead, the generated vendor bundle becomes a tracked asset in the repository
so that `collectstatic` can always pick it up in production.

## In Scope

- add a targeted `.gitignore` exception for `inspinia/static/js/vendors.min.js`
- generate `inspinia/static/js/vendors.min.js` with the existing build
- verify the generated bundle matches the path used by `base.html`
- verify the generated bundle contains the expected vendor stack

## Out of Scope

- switching static assets to a CDN
- changing `STATIC_URL`, `STATIC_ROOT`, `STORAGES`, or WhiteNoise behavior
- adding template-level fallback `<script>` tags
- refactoring DataTables page code
- changing any authenticated page markup

## Implementation Shape

### Repository changes

1. Update `.gitignore` so `inspinia/static/js/vendors.min.js` is not ignored.
2. Run `npm run build` so the existing Gulp pipeline writes the vendor bundle to
   `inspinia/static/js/vendors.min.js`.
3. Commit the generated file alongside the ignore rule change during normal code
   review and deployment flow.

### Why this is the smallest safe fix

- it addresses the confirmed production 404 directly
- it matches the current repo convention of serving compiled static assets from
  the codebase
- it avoids adding new deployment assumptions such as Node-based asset builds on
  the server image

## Validation

Before calling the change complete:

1. Confirm `inspinia/static/js/vendors.min.js` exists after the build.
2. Confirm git no longer ignores that file.
3. Confirm the generated bundle includes the expected vendor libraries from
   `plugins.config.js`.
4. Run `uv run python manage.py check`.

After deployment:

1. Confirm the production `/static/js/vendors.min.js` URL returns `200`.
2. Confirm the broken authenticated page can load the sidenav behavior again.
3. Confirm a representative DataTables page initializes normally in production.

## Rollout

Deployment should remain the normal application deploy:

1. ship the tracked vendor bundle
2. run `collectstatic` as usual
3. redeploy/restart the app
4. verify the static endpoint and affected pages in production

No production config change is required for this fix.

## Risks

### Generated asset drift

Tracking a built bundle means future source changes under `plugins.config.js` or
frontend dependencies must regenerate `inspinia/static/js/vendors.min.js`.
That is acceptable because the repo already expects compiled frontend assets to
live under `inspinia/static/`.

### Possible second-order runtime issues

If production still shows broken UI after `vendors.min.js` is restored, treat
that as a separate bug with fresh runtime evidence. Do not bundle speculative
fixes into this change.

## Acceptance Criteria

The work is complete when:

- `inspinia/static/js/vendors.min.js` is tracked in the repo
- production serves `/static/js/vendors.min.js` without a `404`
- the Inspinia sidenav toggle works again in production
- representative DataTables pages initialize again in production
- no CDN fallback or production settings change was required
